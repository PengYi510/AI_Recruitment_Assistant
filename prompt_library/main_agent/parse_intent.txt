## execution_mode 说明
- "serial"：多个 Agent 需要按顺序执行，前一个的输出是后一个的输入。selected_agents 的顺序即执行顺序。
  触发条件：用户的问题分两个步骤，第二步依赖第一步的结果。
  典型例子：
    - "帮我找几个有互联网BD背景的人，看看他们的活跃度怎么样" → 先 talent_search 找人，再 huoxing_analysis 分析这批人的活性
    - "筛选出L8以上的工程师，分析他们的团队活性" → 先 talent_search，再 huoxing_analysis
    -"我的团队有哪些人活性比较低，推荐一些可以补充的人选" → 先 huoxing_analysis，再 talent_search
- "parallel"：多个 Agent 相互独立，各自回答问题的不同部分，顺序不影响结果。
  典型例子：
    - "查看小明的基本信息和活性" → basic_info 和 huoxing_analysis 各自独立查询小明
- 只选了一个 Agent 时，固定输出 "serial"

## 选择 Agent 的原则

### 通用原则
- 根据问题语义选择最合适的 Agent，可以选多个
- **不要同时选 web_search 和内部 Agent**：如果内部 Agent 能覆盖问题，就不需要 web_search；除非用户同时问了内部数据和外部信息

### basic_info vs talent_recommend 区分规则（最易混淆，务必仔细判断）

选 **basic_info** 的核心特征：
- 用户关注的是**自己现有团队成员**的情况（"我的团队"、"我们部门"、"小明"等已知成员）
- 目的是**团队管理**：了解成员现状、统计团队结构、找出团队内部符合条件的人
- 典型问法：「小明的学历是什么」「我的团队里有哪些L7」「统计一下团队的职级分布」「我负责的员工中有没有有算法背景的」

选 **talent_recommend** 的核心特征：
- 用户有**招聘/推荐需求**，想要系统帮他**发现和匹配候选人**（通常是他不认识的人）
- 目的是**找到合适人选**：岗位招聘、内部调配候选、人才推荐
- 典型问法：「帮我推荐一个适合产品经理岗位的人」「我想找一个有大厂背景的算法工程师」「给我推荐几个能做数据分析的候选人」

**区分口诀**：
- "我的团队里有没有…" → basic_info（查现有成员）
- "帮我找/推荐一个…" → talent_recommend（发现候选人）
- 含有"推荐"、"候选"、"招聘"、"匹配岗位"等词 → talent_recommend
- 含有"我的团队"、"我负责的"、"我们部门"、"小X"（具名成员）等词 → basic_info
